## 架构决策

### 1. 目录结构 - 业务模块化 Monorepo

```
/workspace/
├── apps/
│   ├── ecommerce/
│   │   ├── frontend/    (React + TypeScript)
│   │   ├── backend/     (Node.js + TypeScript)
│   │   └── mobile/      (uni-app + TypeScript)
│   ├── oa/
│   ├── cms/
│   ├── party-building/
│   └── campus/
│
├── types/               (类型定义 - 前后端共享)
│   ├── user.ts
│   ├── order.ts
│   └── index.ts
│
├── docs/                (所有文档)
├── knowledge/           (知识库 - 同步飞书)
└── ...
```

**核心原则**:
- 每个业务独立目录，包含 frontend/backend/mobile
- 类型定义集中放在 `types/`，前后端共享
- 不需要复杂的 packages/shared 结构
- 使用 TypeScript path alias (`@types/*`) 引用类型

### 2. 技术栈

| 层次 | 技术选择 |
|------|----------|
| Web 前端 | React + TypeScript + Vite |
| 移动端 | uni-app + Vue3 + TypeScript |
| 后端 | Node.js + Fastify + TypeScript |
| 数据库 | MySQL + Prisma |
| 缓存 | Redis |
| UI 组件 | shadcn/ui + Ant Design |
| 样式 | Tailwind CSS |
| 测试 | Jest + Playwright |

### 3. 类型共享方案

- 创建独立的 `types/` 目录
- 所有类型定义放在这里
- 前端、后端、移动端都从这里引用
- 使用 tsconfig path alias: `@types/*`
- 修改一处，全局生效

**必须先问的操作**:

**不需要问的操作**:

- ✓ 讨论建议 (不实际修改)
- ✓ 创建记忆文件

### 需求变更管理

**需求分层**:
- L1: 愿景层 (长期不变) - 年变更
- L2: 版本层 (相对稳定) - 月变更
- L3: 功能层 (经常变化) - 周变更
- L4: 任务层 (随时调整) - 天变更

**变更控制流程**:
1. 提出变更
2. 记录到变更日志
3. 评估影响 (工作量/风险)
4. 决定 (本次做/下次做/不做)
5. 执行

**降低变更影响**:
- 早期确认关键决策 (技术栈/数据库/API 规范)
- 模块化设计 (隔离变更影响)
- 测试保护 (变更后快速验证)

## 多 Agent 协作问题诊断

### 遇到的问题

1. **输出和需求有差距**
   - 原因：信息传递链路太长，每层损耗
   - 解决：结构化需求 + Agent 复述确认 + 自动化验收

2. **记忆孤岛**
   - 原因：每个 Agent 有自己的记忆，不互通
   - 解决：LanceDB 共享记忆 + 分层管理

3. **目录混乱**
   - 原因：Agent 乱创建文件，没有规范
   - 解决：PROJECT_STRUCTURE.md + 启动检查清单

### 改进方案

**需求对齐框架 (5 步法)**:
1. 结构化需求文档
2. Agent 复述需求 (确认理解)
3. 任务卡片 (给子 Agent 的指令)
4. 自动化验收
5. 人工验收 + 反馈

**共享记忆**:
- 使用 LanceDB 共享记忆池
- 项目上下文 (只读)
- 需求文档
- 决策记录
- 任务状态
