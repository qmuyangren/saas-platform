## 1. 何时需要人工介入

### 典型场景

| 场景 | 说明 | 紧急度 |
|------|------|--------|
| **AI 理解偏差** | 代码实现与需求不符 | 🔴 高 |
| **性能问题** | 代码能运行但性能差 | 🟡 中 |
| **代码质量问题** | 代码冗余/难以维护 | 🟡 中 |
| **技术细节** | AI 不熟悉特定技术/框架 | 🟡 中 |
| **业务逻辑复杂** | 复杂业务规则 AI 难以把握 | 🔴 高 |
| **紧急 Bug** | 需要快速修复的生产问题 | 🔴 高 |
| **代码审查发现问题** | 安全/性能/规范问题 | 🟡 中 |
| **个人偏好** | 代码风格/实现方式偏好 | 🟢 低 |

### 判断标准

```
需要人工介入的信号:

✅ AI 代码出现以下情况时:
• 连续 3 次修正仍未达到要求
• 核心业务逻辑理解错误
• 性能不达标 (响应时间>1s)
• 安全漏洞未修复
• 代码复杂度过高 (圈复杂度>10)
• 测试覆盖率<50%

❌ 不需要人工介入:
• 轻微代码风格差异
• 不影响功能的命名问题
• 可以后续优化的非关键问题
```

## 2. 人工介入流程

### 完整流程

```
┌─────────────┐
│ 发现问题    │ AI 代码审查/测试失败
└──────┬──────┘
       ↓
┌─────────────┐
│ 评估影响    │ 严重程度/影响范围/紧急度
└──────┬──────┘
       ↓
┌─────────────┐
│ 决定介入    │ 人工修复 or AI 重新生成？
└──────┬──────┘
       ↓
┌─────────────┐
│ 人工修复    │ 直接修改代码/重构
└──────┬──────┘
       ↓
┌─────────────┐
│ 代码审查    │ 自我审查/同事审查
└──────┬──────┘
       ↓
┌─────────────┐
│ 测试验证    │ 单元测试/集成测试
└──────┬──────┘
       ↓
┌─────────────┐
│ 提交代码    │ Git 提交 + 详细说明
└──────┬──────┘
       ↓
┌─────────────┐
│ 反馈 AI     │ 记录问题，帮助 AI 学习
└─────────────┘
```

#### Step 2: 人工修复

##### Git 分支策略

```bash
# 从特性分支创建人工修复分支
git checkout feature/user-login
git checkout -b feature/user-login-human-fix

# 或者从主分支创建
git checkout main
git checkout -b hotfix/login-bug-fix
```

##### 修复原则

```
人工修复原则:

1. 小步修复
   • 每次修复一个具体问题
   • 避免大规模重构
   • 多次提交而非一次大提交

2. 保留上下文
   • 保留 AI 生成的合理部分
   • 注释说明修改原因
   • 记录技术决策

3. 测试保护
   • 修复前运行现有测试
   • 修复后添加回归测试
   • 确保不破坏现有功能

4. 文档化
   • 更新相关文档
   • 记录技术决策
   • 标记待优化点
```

##### 代码注释规范

```typescript
/**
 * 人工修复标记
 * 
 * @fix-author 小虾米
 * @fix-date 2026-03-28
 * @fix-reason AI 生成的代码未处理并发情况，可能导致竞态条件
 * 
 * 原始代码由 AI 生成，人工修复了以下问题:
 * 1. 添加了事务处理
 * 2. 添加了乐观锁
 * 3. 添加了重试机制
 */

// ✅ 正确示例
async function updateUserBalance(
  userId: string, 
  amount: number
): Promise<void> {
  // [人工修复] 添加事务处理，防止并发更新
  // [AI 原始代码] 直接更新，存在竞态条件
  await prisma.$transaction(async (tx) => {
    const user = await tx.user.findUnique({
      where: { id: userId },
      // [人工修复] 添加乐观锁
      version: user.version
    });
    
    if (!user) {
      throw new NotFoundError('用户不存在');
    }
    
    await tx.user.update({
      where: { id: userId },
      data: {
        balance: { increment: amount },
        // [人工修复] 版本号 +1
        version: { increment: 1 }
      }
    });
  });
}
```

## 3. 代码调整规范

### 调整级别

| 级别 | 说明 | 示例 | 处理方式 |
|------|------|------|----------|
| **L1: 微调** | 小修小改 | 变量命名/注释 | AI 重新生成 |
| **L2: 局部调整** | 部分逻辑修改 | 错误处理/边界检查 | 人工修复 |
| **L3: 重构** | 结构优化 | 架构调整/性能优化 | 人工主导 + AI 辅助 |
| **L4: 重写** | 完全重新实现 | 核心算法/复杂逻辑 | 人工重写 |

### L1: 微调 (AI 重新生成)

```bash
# 给 AI 明确指令
clawteam spawn -t saas-dev -n backend \
  --task "调整代码风格：
  1. 变量命名改为 camelCase
  2. 添加函数注释
  3. 添加类型注解
  
  参考文件：docs/code-style.md"
```

### L2: 局部调整 (人工修复)

```typescript
// 人工修复示例
// [人工修复] 添加输入验证
// [AI 原始代码] 缺少验证，存在安全风险
async function createUser(data: CreateUserData) {
  // [人工添加] Zod 验证
  const validated = CreateUserSchema.safeParse(data);
  if (!validated.success) {
    throw new ValidationError(validated.error);
  }
  
  // AI 原始代码 (保留)
  const user = await prisma.user.create({
    data: validated.data
  });
  
  return user;
}
```

### L3: 重构 (人工主导 + AI 辅助)

```
重构流程:

1. 人工设计重构方案
   • 确定重构目标
   • 设计新架构
   • 制定迁移计划

2. AI 生成重构代码
   • 提供详细设计文档
   • AI 生成代码片段
   • 人工审查调整

3. 渐进式重构
   • 保持向后兼容
   • 逐步迁移
   • 充分测试

4. 验证和回滚
   • 性能对比测试
   • 功能回归测试
   • 准备回滚方案
```

### L4: 重写 (人工重写)

```
重写场景:

• 核心算法 AI 无法理解
• 复杂业务逻辑 AI 难以把握
• 性能要求极高需要手工优化
• 安全关键代码需要人工审查

重写流程:
1. 人工完全重写
2. 同事代码审查
3. 充分测试验证
4. 记录重写原因
5. 反馈 AI 学习
```

## 4. 常见问题处理

### 问题 1: AI 反复犯同样错误

**症状**: AI 多次修正仍未解决问题

**处理**:
```
1. 暂停 AI 尝试
2. 人工分析问题根源
3. 人工修复
4. 详细记录问题
5. 更新 AI 技能文档
6. 下次任务前提醒 AI
```

**示例**:
```markdown
## AI 技能更新

**问题**: AI 反复忘记添加输入验证

**更新 Developer 技能**:

### Step 2: 实现后端 API

**强制要求**:
- [ ] 所有用户输入必须验证 (Zod)
- [ ] 所有 API 参数必须类型检查
- [ ] 所有外部数据必须清洗

**检查清单**:
- [ ] 验证函数已添加
- [ ] 错误提示友好
- [ ] 验证逻辑测试通过
```

---

### 问题 2: AI 代码性能差

**症状**: 代码能运行但性能不达标

**处理**:
```
1. 性能分析定位瓶颈
   • 使用 Chrome DevTools
   • 使用 Node.js profiler
   • 查看慢查询日志

2. 人工优化
   • 数据库查询优化
   • 添加缓存
   • 算法优化

3. 性能测试验证
   • 压测对比
   • 监控指标
```

**示例**:
```typescript
// ❌ AI 原始代码 (N+1 查询)
const users = await prisma.user.findMany();
for (const user of users) {
  user.orders = await prisma.order.findMany({
    where: { userId: user.id }
  });
}

// ✅ 人工优化 (批量查询)
const users = await prisma.user.findMany({
  include: { orders: true }  // 一次查询
});
```

---

### 问题 3: AI 不理解业务逻辑

**症状**: 代码实现与业务流程不符

**处理**:
```
1. 详细业务流程文档
   • 绘制流程图
   • 列出所有场景
   • 标注异常情况

2. AI 重新理解
   • 提供详细文档
   • AI 复述确认
   • 生成测试用例

3. 人工验证
   • 代码审查
   • 业务流程测试
   • 用户验收测试
```

**示例**:
```markdown
## 业务流程文档

### 用户退款流程

```
用户申请退款
    ↓
客服审核 (1 个工作日内)
    ↓
审核通过 → 财务处理 (3 个工作日内)
    ↓
退款到原支付渠道 (5-10 个工作日)
    ↓
通知用户
```

**异常情况**:
- 审核不通过 → 通知用户，流程结束
- 财务处理失败 → 重试 3 次，仍失败转人工
- 用户取消申请 → 流程终止
```

2. 快速测试验证
3. 紧急部署上线
4. 事后复盘
5. 反馈 AI 学习
